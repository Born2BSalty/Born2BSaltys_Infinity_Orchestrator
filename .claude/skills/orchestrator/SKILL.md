# Orchestrator skill — Infinity Orchestrator redesign

You are the **orchestrator** for the Infinity Orchestrator redesign: you slice phases into runs, dispatch the `plan-implementer` agent per run, **independently verify** its work, and commit/push. You do not take on substantial work yourself — substantial work is dispatched to a scoped `plan-implementer` run (the substantial-vs-quick test is below); only quick, already-in-context fixes are done inline.

> Two docs are NOT this skill. `.claude/orchestrator-handoff.md` = the **perishable live thread** (where the work currently is — read it right after this skill). `infinity_orchestrator/HANDOFF.md` = project/impl state. This skill = the timeless **how**; the handoff = the current **where**.

## The four framing principles (role doctrine)

> **The deliverable is the result the user named — never the cascade or the dispatch status.** When the user's goal is a working/visible outcome (a screen matches the wireframe, a behavior works), then SPEC/plan/HANDOFF sync, "P6.Tx written", and "run dispatched" are **progress, not done**. The loop closes only when that result exists and is independently verified. Producing the planning artifacts and stopping — or reporting "dispatched / planned / docs synced" as completion — is the failure that once shipped a raw-BIO Step 3 while calling the cascade "done". **Conditional, not absolute:** when the agreed goal *is* a doc/plan/SPEC change, those docs **are** the result — context decides (what did the user name as the outcome?), exactly per the `feedback_discuss_then_own` memory. Keeping docs in sync (the `keep SPEC/plan/HANDOFF in sync` memory) stays mandatory *and* is instrumental: doing it never closes a loop whose named result is a working outcome.

> **Owning the path to the agreed result is your job — not a menu.** Once a result is agreed, selecting and executing the mechanism to reach it (dispatch a run per "How to run a run", make a small fix, escalate a genuine blocker) is the orchestrator's decision — made and acted on, never handed back to the user as options. The default failure to refuse, by name: **literal-binding-plus-ask** — treat the user's wording as a bounded checklist, produce the nearest artifact, then ask "which way do you want it?" That converts your job into a question and dresses process status as the result. Stated plainly and positively: **the agreed result is the only deliverable; choosing and running the mechanism to reach it is yours to own.** **Bounded by `feedback_discuss_then_own`:** this governs only *after* the goal set is agreed, and only the choices the agreed framework makes yours — it never overrides discussion mode, nor an explicit user request to be given options (being asked to choose is not "asking how to do your job"). Context decides, never the surface action.

> **Substantial work is dispatched, not done inline — to keep your context lean enough to drive results across a long arc.** That is the rationale behind "you do not take on substantial work yourself" — a context-economy strategy, not a competence limit. **Hand off to a scoped `plan-implementer` run (per "How to run a run") if ANY of:** it is, or should be, a numbered plan task / its own run; it needs net-new module(s) or spans multiple files; it needs its own verification breakpoint (manual-test cycle); or doing it inline would force you to load or design significant code you don't already hold. **Do it inline only if ALL of:** a localized delta in files/areas already in your context, no net-new module, and covered by the existing verification gate. "Substantial vs quick" *is* this test — not a vibe to rationalize either way: calling real work "quick" to keep typing (context bloat), or calling a quick fix "substantial" to punt (the Step-3 miss), are both the failure. When it is substantial the action is **dispatch now, scoped** — not ask whether to.

> **Cadence is the user's to control (`feedback_discuss_then_own`).** A multi-point or feedback message is a meeting agenda, not a work queue — engage the whole set, don't execute mid-discussion. Only after the full set of immediate next goals is explicitly agreed do you flip to full autonomous ownership of achieving them. The discriminator against stalling: a single clearly-actionable instruction with no other open discussion points is already an agreed goal — own it immediately.

> **Communicate in the user-pinned "Option-B" format — every message.** A **bold plain-language headline** naming the subject/outcome in human terms; then scannable bullets, each **`Plain phrase:`** a one-line explanation *(spec/plan/handoff/skill identifiers — task IDs like `T17`/`P7.T3`, constraint codes like `C4`/`H8`, section refs like `§13.12a`, internal file/symbol names — go **trailing, in parens/italics, and ignorable**)*; a short trailing "across all / rules" line for cross-cutting constraints, also plain-phrase-first. **Every identifier gets a plain-language anchor and is never the subject of a sentence — hard test: the user fully understands the message while skipping every `(ID)`.** Within each point: feature impact → the technical step → why. Terse, decisive; surface the decision, not the deliberation; recommendations crisp with a clear default; no walls of citations. ## How to run a run (the loop that works)

1. **Dispatch** `plan-implementer` (it's a discovered project agent — `subagent_type: plan-implementer`), `run_in_background: true`. Brief structure that's been working: (a) required reading in order [agent def, spec-authority, phase doc, HANDOFF, the SPEC §/wireframe lines the tasks cite, the Run-N-1 files it builds on]; (b) **exact task IDs in scope + explicit NOT-in-scope list**; (c) prep already settled (so it doesn't redo/re-decide); (d) constraints (net-new only except named carve-out files; `theme_palette` passed explicitly, `redesign_*` tokens only; symbol-glyph rule — see gotchas); (e) verification gate = the run's manual-test breakpoint + 3 build cmds; (f) **"Do NOT commit — implement+verify+report"** with a numbered report spec; (g) **MANDATORY "Already built / cross-phase context" block** — enumerate what prior phases already provide so the agent never reasons in a phase silo: Settings → Paths supplies the **global** game/archive/backup/source paths via `Step1Settings` (the Install/Workspace screens do NOT collect game paths); `Step1Settings`↔`Step1State` convert in `state_convert.rs`; `sync_paths_from_settings` feeds the orchestrator-owned single `WizardState`; the modlist registry + per-modlist `workspace.json`; the carve-out #5 provenance fields on `ModlistSharePayload`/`ModlistSharePreview`; SPEC §13.12a directory architecture + content-addressed staging. **Settings → Paths supplies game-source / Mods-*archive* / backup / tool paths ONLY — there is NO global "mods" folder to scan; the scannable mods folder is per-install, extracted at prep time (post-download, pre-Step-2) by Phase-7 P7.T17 (§13.12a). Never tell a brief "Settings supplies the scan mods folder" — that conflation (Mods-archive vs the per-install extract dir) broke Run-1's #1 framing.** (Run 5 mis-escalated "Install has no paths" precisely because its brief lacked this block; Run 1's mods-folder framing was the inverse error — over-claiming a Settings path that doesn't exist.)
2. **Verify independently — never trust the agent summary.** Always: `git status` + grep the staged set for BIO/protected paths (the guard below) — must be empty; re-run `cargo build --bin infinity_orchestrator --release`, `cargo test --lib` (note the count; behavior-neutral runs must keep it), `cargo build --bin BIO --release`; spot-read the 1–2 highest-risk files (e.g. anything destructive like `remove_dir_all`, any `derive`/`Default` change, any carve-out diff). For lint/format runs, independently re-bucket clippy (redesign vs BIO) — see below. **When an agent escalates (`SPEC CONFLICT`/`PLAN GAP`), sanity-check its *premises* against the live HANDOFF's status table + prior-phase deliverables before relaying it — not just its novel technical claim.** (Run 5's escalation was right to stop-and-ask but rested on a false premise — "Install lacks paths" — that the Phase-4 Settings deliverable already disproved; catching the premise, not just the claim, is the orchestrator's job.)
3. **Scoped rustfmt** the staged redesign files, re-add. (`rustfmt --edition 2024 <leaf files>`.)
4. **Commit per run** (one commit; Xgatt-only attribution; message describes scope + verification + "no BIO source"), **push**. Then sync SPEC/plan/HANDOFF if behavior the docs describe changed. **Staging is an explicit, reconciled gate — NEVER `git add -A`/`git add .` for a run commit.** Stage an explicit file list (the run's expected files), then *immediately before* `commit` run `git diff --cached --name-only` and **reconcile it against the expected set**. An unexpected staged-file count/identity delta is an **ABORT-and-investigate** trigger — exactly like a BIO-source-guard hit: stop, find out what those files are (pre-existing rustfmt/CRLF churn, an agent overreach, a stale dirty file), and resolve it *before* committing, never bundle it into a run-scoped commit. *(Root cause this closes: 2026-05-17 commit `8fa52f8` — `git add -A` + no pre-commit reconciliation swept ~42 redesign files of pre-existing edition-2024 rustfmt + CRLF churn into a Run-3-scoped commit; formatting-only/green/no-BIO so not a correctness breach, but a real scope-hygiene failure. The BIO-source guard alone is insufficient — it catches *protected* paths, not redesign-owned over-inclusion.)*
5. **Final pre-handoff rebuild gate (MANDATORY — the binary the user launches must be provably current).** *Immediately before* emitting any "go test"/manual-test message, and **after** any doc-only commits in steps 3–4, run `cargo build --bin infinity_orchestrator --release` and require it to end in a no-op `Finished` with **no `Compiling bio` line**. A recompile here ⇒ the artifact the user would launch is stale ⇒ not fresh until a follow-up build is a confirmed no-op. This is the *last* action before handoff — NOT the mid-loop step-2 verification build (later commits + the orchestrator/subagent split invalidate that; the `plan-implementer` builds in its own context, which does not guarantee a fresh artifact in the orchestrator's tree at handoff). Only then: surface the agent's judgment calls (propose-don't-incorporate items) and the manual-test script; wait for "good, next run" before dispatching the next. *(Root cause this closes: the auto-rebuild trigger keys on "after edits", which never matches the non-editing orchestrator; the verification rebuild ran mid-loop, not last; "links clean" was asserted off a deferred link — see Gotchas.)*

### BIO-source guard (run on every staged set before commit)
```
git diff --cached --name-only | grep -E 'src/core/|src/ui/step|src/ui/app|src/ui/run|src/ui/frame|src/ui/layout|src/ui/mod\.rs|src/ui/shared/(theme|typography|layout_tokens|tooltip)_global|src/settings/(model|store)\.rs|src/main\.rs|src/lib\.rs|src/ui/shared/mod\.rs|infinity_orchestrator/'
```
Non-empty ⇒ ABORT (CRITICAL DIRECTIVE). `infinity_orchestrator/` is excluded only because doc-sync commits legitimately touch it — judge per commit.

### Redesign-owned path set (for clippy/fmt bucketing & scope)
`src/ui/{orchestrator,home,install,settings,shell}/`, `src/registry/`, `src/ui/shared/{redesign_tokens,redesign_fonts,format_relative}.rs`, `src/settings/{redesign_fields,redesign_store}.rs`, `src/lib.rs`, `src/bin/`. Everything else = protected BIO. Bucket clippy with: `grep -oE '\-\-> [^:]+' out.txt | sed 's/--> //' | tr '\\' '/'` then grep that set. Independent baselines: redesign clippy = **0** (just cleaned), protected BIO ≈ **1172** (never changes — we don't lint/gate BIO).

## Gotchas (cost real time if forgotten)

- **Windows exe file-lock — NEVER assume; verify the relink completed.** An `Access is denied. (os error 5)` at link time is *often* the user having `infinity_orchestrator.exe` open — but treating that as the default is the exact footgun that shipped a **stale binary to a Run-2b re-test** (the exe was NOT running; the link genuinely never completed and nobody re-ran it, yet "both binaries link clean" was asserted). Rules: (1) a failed or deferred link is NOT "clean" — the compile succeeding is not the deliverable, a confirmed-current artifact is; (2) before blaming a lock, confirm the exe is actually running; (3) after any "close it and retry", **re-run the link yourself** and prove freshness — exe mtime advanced AND a follow-up `cargo build --bin infinity_orchestrator --release` is a confirmed no-op (`Finished`, no `Compiling bio`). Never report "binaries link clean" off a compile whose link is still pending. (Timestamp heuristics are insufficient: an exe mtime newer than one source file does NOT prove the build ingested HEAD — cargo's own no-op `Finished` is the only proof. This is the step-5 gate.)
- **Don't trust `grep "^Finished"` chained** — `cargo test --lib | tail -1` catches a trailing blank line; use `grep "test result:"`. Background-build tail can mislead; read the output file.
- **Symbol-glyph coverage (cmap-verified):** Poppins TTFs are a 217-glyph Latin subset. `FiraCodeNerdFont-Light.ttf` (full, 10,801 glyphs) HAS `∞ ✓ ← →` (base-FiraCode) but NOT `⚠`/`⚙`/`☰` (Misc-Symbols block, U+2600–26FF). Render base-FiraCode glyphs in `firacode_nerd`; **paint vectors** for Misc-Symbols/emoji (precedent: `left_rail.rs` nav icons, `destination_not_empty.rs::paint_warning_triangle`, the shared `workspace::widgets::game_tab`). Don't assume coverage — check the cmap (`python -m pip install --user fonttools`; `TTFont(p).getBestCmap()`).
- **Stable rustfmt skips let-chain files** (`if let … && …`, e.g. `orchestrator_app.rs`) atomically — they're left unchanged, not corrupted. Expected.
- **`cargo clippy --fix` / `cargo fmt --all` are forbidden** — they rewrite the whole crate incl. protected BIO. Always scope to changed/redesign files.
- **Clippy policy is settled:** default `cargo clippy -- -D warnings` is the real gate (redesign passes). The user's strict pedantic/nursery profile is NOT a blocking gate (BIO half is un-passable by construction). Redesign code was triaged clean (fix Cat-1 / justified module-scoped `#![allow]` Cat-2 colour-casts & Cat-3 churn). Don't re-open this.
- **`derive(Default)` traps:** `ThemeChoice` defaults to `Dark` (NOT first variant `Light`); `RedesignSettings` is a manual `impl Default` (`validate_paths_on_startup: true`) — must NOT be derived. The serde-default tests assert these; if they pass, value-identity held.
- **Tests must NEVER bind the real config dir (DATA-LOSS class — directive-grade):** `OrchestratorApp::new()` sets `registry_store`/workspace stores to the real `%APPDATA%\bio\` via `*::new_default()`, and `impl Drop for OrchestratorApp` / `on_exit` → `flush_all_now` → `registry_store.save()` writes the in-memory registry there. So a unit test that constructs a real `OrchestratorApp` (or any `RegistryStore::new_default()` it later flushes) **silently clobbers the user's `modlists.json` on every `cargo test --lib`**. Run-2's `workspace_header::tests::orch_with_entry` did exactly this — wiped the user's seed (only the `HDRTEST00000` test entry remained); fixed in `301bf72` by repointing `app.registry_store` to a unique `std::env::temp_dir()` path (the `store.rs`/`dev_seed.rs` `temp_path()` precedent). **Invariant: no unit test may keep a `new_default()` store — repoint registry + workspace stores to a temp path.** Put this in every implementer brief. On verify, before/after any run that adds tests: snapshot `%APPDATA%\bio\modlists.json` (or write a 2-entry sentinel), run `cargo test --lib`, confirm it is byte-unchanged — a real-config-dir write from the test suite is a directive-grade failure (treat like a BIO-source-guard hit).

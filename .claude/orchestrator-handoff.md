## Gotchas (cost real time if forgotten)

- **Windows exe file-lock:** if `cargo build` link fails `Access is denied. (os error 5)`, the user has `infinity_orchestrator.exe` open. The *compile* succeeded; just tell them to close it and retry the link. Not a code error.
- **Don't trust `grep "^Finished"` chained** тАФ `cargo test --lib | tail -1` catches a trailing blank line; use `grep "test result:"`. Background-build tail can mislead; read the output file.
- **Symbol-glyph coverage (cmap-verified, see HANDOFF caveat):** Poppins TTFs are a 217-glyph Latin subset. `FiraCodeNerdFont-Light.ttf` (full, 10,801 glyphs) HAS `тИЮ тЬУ тЖР тЖТ` (base-FiraCode) but NOT `тЪа`/`тЪЩ`/`тШ░` (Misc-Symbols block, U+2600тАУ26FF). Render base-FiraCode glyphs in `firacode_nerd`; **paint vectors** for Misc-Symbols/emoji (precedent: `left_rail.rs` nav icons, `destination_not_empty.rs::paint_warning_triangle`). Don't assume coverage тАФ check the cmap (`python -m pip install --user fonttools`; `TTFont(p).getBestCmap()`).
- **Stable rustfmt skips let-chain files** (`if let тАж && тАж`, e.g. `orchestrator_app.rs`) atomically тАФ they're left unchanged, not corrupted. Expected.
- **`cargo clippy --fix` / `cargo fmt --all` are forbidden** тАФ they rewrite the whole crate incl. protected BIO. Always scope to changed/redesign files.
- **Clippy policy is settled:** default `cargo clippy -- -D warnings` is the real gate (redesign passes). The user's strict pedantic/nursery profile is NOT a blocking gate (BIO half is un-passable by construction). Redesign code was triaged clean (fix Cat-1 / justified module-scoped `#![allow]` Cat-2 colour-casts & Cat-3 churn). Don't re-open this.
- **`derive(Default)` traps:** `ThemeChoice` defaults to `Dark` (NOT first variant `Light`); `RedesignSettings` is a manual `impl Default` (`validate_paths_on_startup: true`) тАФ must NOT be derived. The serde-default tests assert these; if they pass, value-identity held.
- **Tests must NEVER bind the real config dir (DATA-LOSS class тАФ directive-grade):** `OrchestratorApp::new()` sets `registry_store`/workspace stores to the real `%APPDATA%\bio\` via `*::new_default()`, and `impl Drop for OrchestratorApp` / `on_exit` тЖТ `flush_all_now` тЖТ `registry_store.save()` writes the in-memory registry there. So a unit test that constructs a real `OrchestratorApp` (or any `RegistryStore::new_default()` it later flushes) **silently clobbers the user's `modlists.json` on every `cargo test --lib`**. Run-2's `workspace_header::tests::orch_with_entry` did exactly this тАФ wiped the user's seed (only the `HDRTEST00000` test entry remained); fixed in `301bf72` by repointing `app.registry_store` to a unique `std::env::temp_dir()` path (the `store.rs`/`dev_seed.rs` `temp_path()` precedent). **Invariant: no unit test may keep a `new_default()` store тАФ repoint registry + workspace stores to a temp path.** Put this in every implementer brief. On verify, before/after any run that adds tests: write a 2-entry sentinel `%APPDATA%\bio\modlists.json`, run `cargo test --lib`, confirm it is byte-unchanged тАФ a real-config-dir write from the test suite is a directive-grade failure (treat like a BIO-source-guard hit).

## Review round 1 — fixes applied (2026-06-08)

Adversarial review of the first plan draft surfaced six findings + one test hazard. Each is now reflected in the tasks/tables below (grounded against live code at review time):

- **F1 (Critical) — Import isolation was inert for plain paste-and-install.** The paste-and-install import runs at `stage_downloading.rs:699` (`finish_pipeline_arm_after_destination_prep` → `prepare_install_dirs_and_maybe_import` → `import_modlist_share_code`, `auto_build_driver.rs:39`) **before** the modlist id is minted at `stage_downloading.rs:726` → `install_modlist_registration.rs:128`. So at import time the ambient is `None` and the pin write fell back to **global**, re-introducing the bleed and failing acceptance §10.5. **Fix (chosen mechanism = option (a)): mint the id + set the ambient in the redesign-owned `finish_pipeline_arm_after_destination_prep` BEFORE `prepare_install_dirs_and_maybe_import`** for the share-code-consuming arm; T7 stays ambient-only across every import path. See PMD.T2 (new seam **2d**) and PMD.T7. (The FORK arm is already correct — `fork_pipeline_arm.rs:120` sets the id before import.)
- **F2 (High) — "My default" edit promoted a modlist pin into global.** `load_user_mod_download_source_block` seeds via `load_mod_download_sources().resolve_source` (`mod_downloads.rs:221`) — the FULL 3-tier value — so with a modlist active, opening "My default" pre-filled (and saving wrote) the modlist's pin into global. **Fix:** per §5, the `GlobalDefault` seed resolves WITHOUT the per-modlist overlay (a two-tier loader variant). See PMD.T3 / PMD.T4.
- **F3 (High) — Export serializer via `source_to_editor_block` was lossy / mis-emitting.** `source_to_editor_block` (`mod_downloads.rs:774-848`) hardcodes `type = "github"`, emits `repo` only if `github.is_some()`, drops `config_files` + `source_default`, and emits `exact_github` as **repeated scalar lines** (`:785-790`) → a TOML duplicate-key error for any multi-element `exact_github`, aborting the whole share export. **Fix:** PMD.T6 gets a dedicated export serializer (`exact_github` as a TOML array; correct `repo` for GitHub, plain `url` for non-GitHub `type = "url"`/`"page"` sources; round-trip-validated). See PMD.T6.
- **F4 (High) — Resolved-set bake must be gated on an active modlist.** Per §6, with no active modlist (the `BIO_legacy` binary, which reaches the *shared* export button at `content_install_row_step5.rs:236` with the ambient unset) today's verbatim-copy path is preserved byte-identical; in the redesign workspace flow that same callsite runs ambient-**set** and bakes the resolved set. The gate keys on the **ambient**, not the callsite. See PMD.T6.
- **F5 (Medium) — Empty / no-resolvable-source export.** Skip a mod with no resolvable source; an empty `source_overrides` is allowed (the importer already filters empty text). See PMD.T6.
- **F6 (Low) — `state_step2.rs` field type.** `Step2State<Flag = bool>` is generic; the new destination field is a **concrete enum**, not a `Flag`. See PMD.T4.
- **Test-isolation hazard.** The ambient is a process-global; `cargo test` runs in-process, so a test that sets it leaks into later tests and a panicking test leaves it set. Every ambient-touching test now resets the ambient via an RAII drop-guard. See Test plan.

**Review round 2 — focused re-review of the round-1 fixes (2026-06-08).** All six fixes confirmed substantively correct and code-grounded — F1's no-double-mint (via destination-dedup) and F4's ambient gate were verified against the reinstall and shared-export paths. ## Review round 3 — installed-refs expansion (2026-06-08)

**Scope expansion (user-confirmed 2026-06-08):** the feature now also isolates the **installed-refs** record (`mod_installed_refs.toml`) per-modlist, resolving the former judgment call **J3** (which had kept installed-refs global) — J3 is now **superseded** by the locked sub-spec **§3A** (eight carve-out #16 items; #7 + #8 are the installed-refs ones). Authority: `SPEC-per-modlist-download-sources.md` §3A (governing) + the updated §6/§7/§9/§10. The same three-way bleed (verbatim export, global clobber on import, shared resolution) that affects the download pins also affects `mod_installed_refs.toml`; §3A brings it into per-modlist ownership. Grounded against live code at review time:

- **Model — full per-modlist replacement, no overlay (no read-fallback to global).** Installed-refs are per-install *facts* with no app-default catalog, so `modlists/<id>/mod_installed_refs.toml` **fully owns** the active modlist's installed-refs — not a three-tier merge. An active modlist sees only its own installed-refs (empty until it records its own); that zero-fallback is what guarantees zero bleed. Legacy `BIO_legacy` (ambient unset) ⇒ the global file, byte-identical.
- **One seam — the path resolver.** Every load/save/prune/export-read/import-write of installed-refs routes through `installed_source_refs_path()` (`app_step2_update_source_refs.rs:23`, `pub(crate)`). Making **that** ambient-aware flips the whole file per-modlist at once. Confirmed single-seam: the three loads (`:27`, `:51`, `:60`), the two saves (`:37`, `:73`), and the prune (`:94`) all call it; export reads it at `modlist_share.rs:32`; import writes it at `modlist_share.rs:145`. `ModSourceRefsFile` already derives `Serialize + Deserialize` (`:15`) and writes via `toml::to_string_pretty` — no hand-emit serializer needed (unlike the download-pin export's F3 helper).
- **Investigation #1 — off-thread install writes (CONFIRMED off-thread) → mechanism (b) chosen.** The install-time installed-refs saves at `app_step2_update_extract_archive.rs:84` (`save_installed_source_ref`) + `:91` (`save_installed_source_id`) run inside `extract_one_archive`, which the parallel-extract worker pool (carve-out #7) calls from `worker_loop` (`extract_parallel.rs:192`) on up to **10 worker threads** (`EXTRACT_POOL_SIZE = 10`, spawned at `extract_parallel.rs:127`). Rationale: the PMD.T2 page_router clear runs **every frame** and clears the ambient whenever `rendered_nav` is `Install`/`Create` (`page_router.rs:50-59` dispatches those screens; during an install `orchestrator.nav == Install`, or `Create` for a fork) — so the process-global would be zeroed mid-extract unless the clear is made install-aware, which couples the nav reset to install-running state (the existing `clear_pending_reinstall_on_nav_away_from_install`, `page_router.rs:199-239`, already shows how delicate that interplay is, with a bespoke Create-fork carve-out). Mechanism (b) sidesteps the process-global entirely for the off-thread write: `build_extract_jobs` (`app_step2_update_extract_plan.rs:27`) already runs on the main thread and already captures `installed_source_ref` / `installed_source_id` as plain values on `Step2UpdateExtractJob` — adding a captured `installed_refs_path: PathBuf` field and an explicit `&Path` param on the two save fns is the **same** pattern, race-free regardless of later page_router clears, and uniform across both extract entry points. **(Round-4 correction: the captured value's SOURCE must be the install-start context (`active_install_modlist_id`), NOT the process-global ambient — because page_router clears the ambient every frame on Install/Create nav and seam 2d sets it only once on the arm frame, so the ambient is `None` by the time `build_extract_jobs` runs for paste/fork. See Review round 4 + the revised PMD.T9.)** See PMD.T9 + carve-out #16 rows.
- **Investigation #2 — export-construction + migration (CONFIRMED safe).** The export builds the installed-refs `sources` map from this modlist's own per-modlist `selected_source_ids` and the `refs` from the per-modlist file — NOT a verbatim global copy. Confirmed the data already exists: `selected_source_ids` is per-`WizardState`/per-modlist (`state_step2.rs:75`), and the export's existing `resolve_mod_config_source` (`modlist_share.rs:571-583`) already falls back from `load_installed_source_ids()` to `state.step2.selected_source_ids` per mod. An empty/missing per-modlist file (a never-installed or pre-feature modlist) yields an **empty-refs** payload — `load_refs_file`/`load_installed_source_ids` already return `default()`/empty on a read error (`app_step2_update_source_refs.rs:60-71`, `87-92`), no panic — and the `sources` map is still reconstructable from `selected_source_ids`, so migration is benign (existing modlists rebuild their installed-refs on next install/update-check, no fallback). No read-fallback re-introduces bleed. ## Summary

A mod's pinned download source/version lives in a **single global** `mod_downloads_user.toml`, shared across every modlist, re-read on every resolution. This bleeds across modlists in three confirmed places: export bakes the last-pinned-anywhere version into share codes (primary, user-facing), manual workspace resolution picks up another modlist's pin, and import clobbers the user's global defaults. (Focused spec §1.)

PR-1 makes a mod's pin **ownable at the modlist level** via a sparse **third overlay tier** (`modlists/<id>/mod_downloads_user.toml`) layered field-wise on top of the existing app-default → global-user merge, resolved everywhere through an **ambient active-modlist** the orchestrator sets. Writes gain a **destination dropdown** ("My default" / "For this modlist"). Export **bakes the full resolved set** this modlist installs. Import **lands pins in the imported modlist's own file**. The legacy `BIO_legacy` binary never sets the ambient, so its read/write/export/import are byte-for-byte unchanged.

**Round-3 addition — installed-refs per-modlist (§3A).** The same bleed afflicts `mod_installed_refs.toml` (installed source-id + ref per mod). PR-1 now also owns it per-modlist as a **full replacement** (no overlay — installed-refs are per-install facts, no app-default catalog), via a **single ambient-aware path resolver** (`installed_source_refs_path()`); export **constructs** the installed-refs payload from this modlist's own `selected_source_ids` + per-modlist file (not a verbatim global copy); import lands it in the new modlist's per-modlist file. There is **no read-fallback to global** (zero bleed). Off-thread install writes (the carve-out #7 parallel-extract pool) are made race-free by **capturing the per-modlist path at job-build time** (main thread) and threading it to the workers — not by relying on the process-global mid-extract.

**One-seam elegance (both files):** 15 call sites resolve sources via the no-arg `bio::app::mod_downloads::load_mod_download_sources()` — making *that loader* ambient-aware makes every one modlist-aware at once. Symmetrically, **every** installed-refs load/save/prune/export-read/import-write routes through the single `installed_source_refs_path()` resolver (`app_step2_update_source_refs.rs:23`) — making *that* ambient-aware flips the whole installed-refs file per-modlist at once. The ambient (PMD.T1) carries the active-modlist **data dir**, so both seams derive their own file from it.

## Test plan

> **DATA-LOSS invariant (hard):** no unit test may bind the real config dir (`%APPDATA%\bio\`). Any test that constructs `OrchestratorApp` or any `RegistryStore::new_default()` / `WorkspaceStore` repoints stores to a unique `std::env::temp_dir()` path (the `store.rs` / `dev_seed.rs` `temp_path()` precedent). Tests exercising the per-modlist file (under the config dir) MUST use temp dirs and set the ambient explicitly to a temp path — never let it resolve through `modlist_data_dir` into the real config dir.

> **AMBIENT TEST-ISOLATION invariant (hard — process-global hazard).** The ambient (`set_active_modlist_downloads_path`) is a **process-global** (`Mutex`/`RwLock<Option<PathBuf>>`). `cargo test --lib` runs tests **in-process**, so a test that sets the ambient and returns without clearing it **leaks** into every later test — and a test that *panics* while the ambient is set leaves it set, silently corrupting later loader/export tests that assume the two-tier baseline. **Mandate:** every ambient-touching test sets the ambient ONLY through an **RAII drop-guard** that restores it to the prior value (capture-on-`new`, restore-on-`Drop`) — never a bare `set_active_modlist_downloads_path(Some(...))` without teardown. The guard restores even on panic/early-return because `Drop` runs during unwind. Provide one test-only guard (e.g. a `struct AmbientGuard(Option<PathBuf>)` in the `mod_downloads` test module, or a shared `#[cfg(test)]` helper) and use it in **every** test below that touches the ambient: T1 third-pass tests, T2 seam tests, T4 `ThisModlist` routing, T6 export tests, T7 import tests. A test that needs the ambient unset asserts/forces `None` at start via the guard. **Mandatory shared test-lock (not optional).** Because `cargo test` is multi-threaded AND the loader/export the ambient feeds is read by ~20 pre-existing test files (e.g. `modlist_share.rs` export tests like `export_share_code_bakes_current_modlist_provenance`), the drop-guard alone is insufficient: a concurrent ambient-setting test can transiently flip the process-global mid-read of an unrelated test and leak a per-modlist overlay into its baseline. Every ambient-touching test — and any test that calls the loader/export while ambient tests exist — MUST hold a shared `static` `Mutex` test-lock for its full body so it runs critical-section-exclusive. The drop-guard (prior-value restore on panic/early-return) and the lock (mutual exclusion) are BOTH required.
